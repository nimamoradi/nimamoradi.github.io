Shortcomings of Android DRM Wrappers

Android applications can be easily installed without using a marketplace, which poses a risk for mass distribution of paid apps. To counter this, stores and application developers implement various measures to safeguard apps. However, most available wrappers are insufficient for protecting the APK file. After a year of designing and developing wrappers for DRM management and statistics reports, here are some problems I observed.


Let's examine how some stores enforce DRM.

Amazon Appstore

The Amazon Appstore, one of the oldest Android app stores, hosts about half a million different applications for Android phones, tablets, and Fire tablets. Despite its long-standing presence, its protection method for pre-paid and free applications is surprisingly simple to crack.

For testing purposes, I purchased an app and noticed that if the Amazon Appstore is not installed, the program prompts with a dialog to install it.

Extracting the APK

First, we should extract the apk you can use adb to find the location.

adb shell pm path com.example.someapp

then pull the apk

adb pull /data/app/com.example.someapp-2.apk path/to/desired/destination

or use many free app extractors that are available(which are doing something similar).

Decompiling the APK

Next, decompile the APK using apktool:

apktool d  -f --use-aapt2 -v  -o GAME_APK_FILE GAME_APK_EXTRACTED_DIST

The two common methods that wrappers use on Android involve changing the initial activity to provide subscription checks or modifying the parent class of an activity or application.

Let's examine the AndroidManifest.xml file.

Manifest Examination

Here is the application class.

<application android:name="androidx.multidex.MultiDexApplication">
    <activity android:name="com.company.not.aws" >
        <intent-filter>
            <action android:name="android.intent.action.MAIN"/>
            <category android:name="android.intent.category.LAUNCHER"/>
        </intent-filter>
    </activity>
</application>

As you can see, Amazon has not changed the initial activity class or added/swapped its own.

Let's check if any activity parent has changed or if code has been injected:

.class public Lcom/company/app/Activity;
.super Lcom/amazon/android/activity/AmazonActivity;

Here, Amazon has injected code and changed the activity class to inherit from its own activity.

By removing the functionality of the dialog and checking mechanisms, the app can be left unprotected, which at this point is easy to achieve.

You can then rebuild the app using apktool.

Another Wrapper Mistake

I encountered another example where I cannot disclose the company's name, but their services were for many mobile network providers in Europe, serving millions of users daily.

<activity android:configChanges="keyboardHidden|orientation" 
    android:name="com.***.***.old_launcher" 
    android:screenOrientation="portrait" 
    android:exported="true"
/>

In this case, the developer replaced the original app launcher but forgot to set exported to false, allowing the activity to be bypassed without rewrapping. This means you can call the activity from other apps if it's exported.

Conclusion

In summary, while DRM wrappers are designed to protect Android applications, many current implementations fall short of providing robust security.
